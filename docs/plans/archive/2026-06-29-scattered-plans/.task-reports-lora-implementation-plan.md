# Planner Report

## Status
split-required

## Why Split / Parallelize
IC-LoRA product completion touches backend download/recommendation, backend request/pipeline plumbing, frontend workflow UI/state, generated API types, tests, and live smoke. These are sequential because API/schema changes feed frontend types and all phases share IC-LoRA validation state. Keep each phase narrow; do not touch the current ImageConditioner skip-encoder fix because it is being fixed separately.

## Interference Check
- parallel safe: no
- shared files or generated outputs: `backend/api_types.py`, `frontend/generated/backend-openapi.ts`, `backend/tests/test_ic_lora.py`, model profile/download state
- shared validation state: Python typecheck/tests, TS typecheck/build, live model cache/downloads
- worktree isolation required: no, but sequential phases required
- rationale: backend API changes must land before UI typing; download recommendation behavior affects UI availability; live smoke must run after ImageConditioner skip-encoder fix is present.

   - purpose: make all official IC-LoRA adapter files, depth processor, HDR companion embeddings, and required original LTX assets visible to status/recommendation/download flow.
